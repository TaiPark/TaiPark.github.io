    存储的问题

  • 链表
    • 插入删除很快
    • 若查找的元素在中部、尾部，需要将链表中的元素全部遍历 => 速度慢
  • HashMap
    • Hash碰撞，jdk1.8之前是头插法，之后尾插法
    • 随着碰撞的越来越多，链表存储越来越长，jdk1.8后进行升级，大于8的时候变成红黑树 => 速度快
    • 数组结构的存放模式能够通过下标快速获取数据
    • 不能满足海量数据的存储，因为太大了内存放不下，1万个int占用了1M的空间，10亿的数据需要10万兆

  • BitMap

    • 存储1 3 5 7：
      用int存储 4个int
      用bit存储 1byte

    • BitMap的思想：天然有序的状态，解决乱序自然数排序的问题

    • 存储 1 3 5 7E
      随着集合内最大元素的增大而增大，此时消耗的空间远大于基本的数据结构
    • 100W用户，1台MySQL统计日活，如何去统计？
      用户有不重复的ID，若ID是数字可以通过位图法，只要登录了把用户ID的位置改成1.只要统计1的数字值即可得到日活用户
  • 布隆过滤器
    • 需求：在亿级URL中寻找黑名单
    • 布隆过滤器底层是一个二进制的数组结构，当有元素进来会变成1
    • URL地址会经过多个hash()，拿到数字下标，并修改 => 多个hash增大随机性，减少hash碰撞的概率 => 数据增加误判率增加
    • hash()势必会有哈希碰撞，不存在的值可能经过hash()拿到的值都为1 => 误判 => 判断存在的不一定存在，判断不存在的一定不存在
    • 随着位数组的长度越来越长，误判率越来越低
    • 存放的不是完整的数据，占用内存少，且新增查询够快，删除很麻烦 => 有位置共享，若删除置0，其他的元素去读可能就当做不存在了 => 第一种删除方法：用计数器映射，有元素来+1，删除-1（一般不用） => 第二种方法：不管，使用布隆过滤器需要元素的更改不是很频繁（new一个布隆过滤器然后替换）
    • Google Guava包：提供布隆过滤器（传入预计插入数据与预计误判率），原生数据的各种操作。

  • Redis的BitMaps

    • 统计日活月活

    • 用户登录时update +1，是非关键性任务（热点数据问题），单节点的MySQL数据库并发比较好的是500-700，仅5000个人登录需要10台MySQL => 解决问题可以使用缓存数据库，用Redis分摊压力 => Redis并发高

      UUID在位图法中处理 => 1、二进制表示 => 2、做一个不重复的数字ID的映射

    • bitcon寻址过程通过布隆过滤器可以做到
    • Redis为什么要实现BitMap？
      缓存穿透、缓存雪崩、缓存击穿的问题
      缓存雪崩：缓存雪崩是指缓存中数据大批量到过期时间，而查询数据量巨大，引起数据库压力过大甚至down机。和缓存击穿不同的是，缓存击穿指并发查同一条数据，缓存雪崩是不同数据都过期了，很多数据都查不到从而查数据库。 => 用setnx(k,v),60s，若有10W条数据都是60s => 60s后数据过期，而Redis中没有数据了，MySQL就需要承载压力导致宕机
      缓存穿透：缓存穿透是指缓存和数据库中都没有的数据，而用户不断发起请求，如发起为id为“-1”的数据或id为特别大不存在的数据。这时的用户很可能是攻击者，攻击会导致数据库压力过大。 => 发送Redis中没有的数据，Redis会去MySQL中找，导致数据库宕机 => 在Redis前面加一个布隆过滤器，请求来的时候先用其过滤，若是有的话再走正常的流程
      缓存击穿：缓存击穿是指缓存中没有但数据库中有的数据（一般是缓存时间到期），这时由于并发用户特别多，同时读缓存没读到数据，又同时去数据库去取数据，引起数据库压力瞬间增大，造成过大压力。
    • Redis逻辑思路
      1.MySQL中哪些是热点数据？
      2.数据结构的问题？
      3.如何大量导入Redis？ => pipeline管道模式
      4.Redis测试，有没有达到解决热点数据的问题？ => 慢查询优化 => 上线之前进行压测 => value越大，并发能力越差 => Redis大value的问题 => 大于10k的KV就是大value，影响并发性能 => 用户并发到某个Key时如何抗住？
      5.单节点的Redis变成集群方法 => 主从、哨兵、Cluster
      6.数据一致性问题

      补充：

  • 数据的离散性：选择性越多优化越差，如果在一列加上索引需要看一下当前一列不重复的值是多少，总共有多少值，进行比例越大，比例越大代表离散性越好
  • like百分号加到前面：所有元素都要看，索引是无用的
  • like百分号加到后面：有了一定的选择性
